# 实战示例

**场景：** 检查代码库中 20 个 API 端点是否都有对应的请求参数校验

**分析：**
- 每个端点的校验检查是独立的判定
- 不需要了解项目全局上下文
- 返回"有/无校验"即可
- 满足判定型准入条件

**Dispatch：**
```
子代理 1 → 检查 /api/users POST 是否有参数校验
子代理 2 → 检查 /api/users GET 是否有参数校验
...
子代理 20 → 检查 /api/orders DELETE 是否有参数校验
```

**结果：**
- 18 个返回"有"
- 2 个返回"无"（具体端点名）

**聚合：** 主代理汇总，直接定位到需要补校验的 2 个端点

---

**如果错误地拆分会怎样：**

如果把"检查校验"拆成"分析每个端点的业务逻辑 → 判断是否需要校验 → 检查是否有校验"，每个子代理需要理解业务逻辑，返回分析结论，主代理还要综合判断——这违背了准入条件，结果比串行更慢。
