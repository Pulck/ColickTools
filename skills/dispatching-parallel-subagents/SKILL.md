# 并行派遣子代理

## 核心目的

为主代理提供**拆分后的执行标准**，不替代主代理的拆分决策。上下文独立性是唯一准入条件。

**规范 ≠ 限制。** 本 Skill 提供并行拆分的安全框架，不是束缚 AI 能力的枷锁。当推荐性规范与你的判断冲突时，你的判断优先；核心原则层面的冲突需先与用户沟通。

## 责任边界

**主代理的职责：**
- 分析任务是否可拆（接口锁定、模块独立、上下文可隔离）
- 设计接口契约和验收标准
- 决定是否拆分为并行子代理

**本 Skill 的职责：**
- 提供拆分后的执行规范（prompt 结构、返回格式、聚合策略）
- 明确准入条件和禁止事项
- 提供常见错误的反面示例

## 准入条件

**必须同时满足：**
- 子任务之间不需要交换中间结果
- 子任务不需要继承主会话历史
- 每个子任务可自闭执行、返回无歧义结果

**不满足时，保留在主会话串行执行。**

## 子代理类型

### 判定型
- **任务：** 独立查询验证（存在性检查、匹配提取）
- **输入：** 关键词 + 判定标准
- **返回：** 是/否、行号、无匹配（一句话，无解释）

### 实现型
- **任务：** 接口已锁定的独立模块编码
- **输入：** 接口契约 + 验收标准 + 约束清单
- **返回：** DONE/ERROR/BLOCKED + 产物 + 自测结果

## 不可违背的原则

1. **子代理之间不得交换中间结果** — 破坏上下文隔离，token 损失超过并行收益
2. **子代理不得继承主会话历史** — 失去并行拆分的意义

## 推荐性规范

- 独立性是准入门槛，不满足则不拆
- 返回应当无歧义
- 子代理应自治，主代理不介入内部过程
- 实现型子代理内部串行完成（架构→编码→测试→review），不再拆分

## 聚合策略

- **判定型：** 主代理直接汇总布尔结果；>10 个时分批
- **实现型：** 主代理集成验证；>5 个模块时分批集成

## 失败处理

1. 子代理失败 → 结束当前子代理
2. 派遣全新子代理，仅携带最小上下文，不继承纠缠历史
3. 修正子代理仍失败 → **停止并行，在主会话串行处理**

## 突破规范前的沟通原则

以下情况**必须**先与用户讨论：
- 需突破核心原则（子代理间交换结果、继承主会话历史）
- 原设计的模块接口或拆分方式无法成立
- 同一问题已尝试 2 次以上仍未解决
- 你的判断与用户的显式指令冲突

**原则性要求 ≠ 执行指令。** 用户说"先讨论""你觉得呢"是邀请讨论，不是授权执行。
