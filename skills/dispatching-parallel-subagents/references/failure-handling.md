# 失败处理与常见错误

## 何时放弃子代理、回到主会话

如果修正子代理仍然失败，通常说明以下至少一项成立：

- 任务边界定义不清，实际依赖了未预见的全局上下文
- 接口假设不成立，模块间存在隐性耦合
- 问题本身需要连续推理，不适合隔离执行

此时建议停止并行，在主会话串行处理。强行继续派遣子代理通常只会增加 overhead 而无收益。

## 常见错误

**❌ 为并行而并行：** "这个任务很大，拆成 5 个子代理吧" — 大不等于独立
- 常见想法："不拆分上下文会爆"
- 现实：强行拆分有依赖的子任务，token 损失反而更大
- **正确：** 先检查每个子任务是否满足准入条件，不满足就不拆

**❌ 传递全局上下文：** 在 prompt 里附上"项目背景""之前做过什么"
- **正确：** 只给该子任务需要的关键词和判定标准

**❌ 返回分析型结果：** "这个函数存在，但实现方式不太优雅，建议重构..."
- **正确：** "存在" — 一句话，无歧义

**❌ 实现型子代理内部再拆：** 模块 A 的子代理又把编码和测试拆给两个子代理
- 常见想法："子代理有能力，应该让它自主决定"
- 现实：子代理的自主仅限任务内部，核心实现不应再拆
- **正确：** 模块内部串行完成，只有独立判定任务可派二级子代理

**❌ 集成时反向要求修改：** "模块 A 的接口和 B 不匹配，A 你改一下"
- 常见想法："模块接口差不多定了，可以边做边调整"
- 现实：接口未锁定 = 不满足实现型准入条件
- **正确：** 子代理已结束，由主代理处理或重新派遣修正子代理

**❌ 返回半成品：** "编码完成，测试写了一半，需要你帮忙看看测试怎么写"
- 常见想法："先拆试试，不行再回主会话"
- 现实：拆分的 overhead 本身就是成本，试错成本高
- **正确：** 返回 DONE + 完整产物 + 自测通过结果
