# 实战示例

**场景：** 检查代码库中 20 个 API 端点是否都有对应的请求参数校验

**分析：**
- 每个端点的校验检查是独立的判定
- 不需要了解项目全局上下文
- 返回"有/无校验"即可
- 满足判定型准入条件

**Dispatch：**
```
子代理 1 → 检查 /api/users POST 是否有参数校验
子代理 2 → 检查 /api/users GET 是否有参数校验
...
子代理 20 → 检查 /api/orders DELETE 是否有参数校验
```

**结果：**
- 18 个返回"有"
- 2 个返回"无"（具体端点名）

**聚合：** 主代理汇总，直接定位到需要补校验的 2 个端点

---

**如果错误地拆分会怎样：**

如果把"检查校验"拆成"分析每个端点的业务逻辑 → 判断是否需要校验 → 检查是否有校验"，每个子代理需要理解业务逻辑，返回分析结论，主代理还要综合判断——这违背了准入条件，结果比串行更慢。

---

**场景：** 为 CLI 工具并行实现 3 个独立的子命令模块

**主代理的前期工作（拆分前）：**
- 统一设计命令注册接口：`registerCommand(name, handler, options)`
- 锁定参数解析格式：`{ input, output, verbose }`
- 确定错误处理模式：统一抛 `CliError(code, message)`
- 确认 3 个命令之间无依赖：`validate`、`export`、`sync`

**Dispatch：**

```
子代理 1 → 实现 validate 命令：读取配置文件并校验格式
  输入：接口契约 + 验收标准 + 约束清单
  返回：DONE + 代码 + 自测结果

子代理 2 → 实现 export 命令：将数据导出为 JSON/CSV
  输入：接口契约 + 验收标准 + 约束清单
  返回：DONE + 代码 + 自测结果

子代理 3 → 实现 sync 命令：与远程服务器同步状态
  输入：接口契约 + 验收标准 + 约束清单
  返回：DONE + 代码 + 自测结果
```

**结果：**
- 3 个均返回 DONE，产物为可直接集成到 `src/commands/` 的完整模块文件
- 每个产物包含单元测试，且均通过

**聚合：** 主代理按统一接口集成到命令注册体系，运行集成测试验证命令行解析和错误处理一致性

---

**如果错误地拆分会怎样：**

如果接口未锁定就拆分——例如子代理 1 把 `validate` 的返回格式设计成 `{ valid, errors }`，而主代理预期的是布尔值——集成时会发现不匹配。此时若反向要求子代理 1 修改，违背了"接口已锁定"的准入条件。正确做法是停止并行，在主会话重新设计接口后再拆。
